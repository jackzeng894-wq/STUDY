# 系统需求分析文档

## 1. 引言

### 1.1 背景

在数字化与智能化深度融合的时代，高等教育正从"标准化教学"向"个性化学习"转型。《中国教育现代化2035》明确提出要构建智能化、个性化的教育体系。然而当前教学实践中，以下矛盾日益突出：

- **资源供需错配**：互联网存在海量学习资源，但学生筛选成本极高，大量时间消耗在"找资料"而非"学知识"
- **教学模式固化**：课堂讲授采用统一节奏，无法适配学生个体差异。同一课堂上，有的学生"吃不饱"，有的学生"跟不上"
- **反馈链路断裂**：学生做完练习后缺乏即时、精准的诊断反馈，错误认知得不到及时纠正
- **传统辅助工具能力不足**：现有学习平台（慕课、题库）本质上是"内容仓库"，缺乏生成能力、分析能力和个性化适配能力

### 1.2 项目目标

构建一个**基于大模型的多智能体个性化学习系统**，以JavaScript程序设计课程为切入点，实现：

1. **对话式学生图像自动构建**：通过自然语言对话自动提取学生特征，构建6维度动态画像，替代传统表单填写
2. **多智能体协同资源生成**：由不同角色的AI智能体协作，为学生生成适配其水平的6种类型个性化学习资源
3. **知识图谱驱动学习路径规划**：基于图算法（拓扑排序、最短路径、PageRank）为每位学生规划最优学习路径并动态调整

### 1.3 适用范围

- 主要用户：高校计算机及相关专业学生，尤其是正在学习或复习JavaScript课程的本科生
- 次要用户：授课教师（通过系统了解班级学情画像）
- 应用场景：课后自学、考前复习、查漏补缺、跨专业自学

## 2. 用户需求分析

### 2.1 用户分层与画像

#### 用户画像一：编程零基础新生（大一）

| 维度 | 描述 |
|------|------|
| 典型情况 | 计算机专业大一新生，刚接触编程，第一门语言是JavaScript |
| 知识基础 | 无编程经验，对变量、循环等基本概念无认知 |
| 学习痛点 | 课堂节奏快，跟不上；教科书术语太多看不懂；不知道从哪里开始 |
| 核心需求 | 需要有人告诉他"先学什么后学什么"，需要通俗易懂的入门资料 |
| 行为特征 | 愿意跟着系统引导走，偏好图文并茂、有例子可抄的教程 |
| 时间投入 | 每周6-8小时 |

#### 用户画像二：有编程基础但跨语言学生（大二）

| 维度 | 描述 |
|------|------|
| 典型情况 | 大二学生，学过C/C++或Java，选修Web前端方向 |
| 知识基础 | 理解变量、函数、循环等基础概念，但不熟悉JavaScript特有语法（原型链、闭包、事件循环） |
| 学习痛点 | 用C语言的思维写JS代码，踩坑多；不理解的几个特定概念反复绊倒 |
| 核心需求 | 需要对比式教学（"JS的this跟Java的this有什么不同"），需要针对性的错题练习 |
| 行为特征 | 有明确的学习目标，知道自己的薄弱点，喜欢看代码示例和实战案例 |
| 时间投入 | 每周4-6小时 |

#### 用户画像三：进阶备战学生（大三/大四）

| 维度 | 描述 |
|------|------|
| 典型情况 | 大三学生，学完了JS课程，准备校招面试或做毕业设计 |
| 知识基础 | 大部分知识点已掌握，但存在知识盲区；对框架（React/Vue）有一定了解 |
| 学习痛点 | 不知道自己还有哪些薄弱点；面试题刷了很多但不成体系 |
| 核心需求 | 快速扫描知识盲区，针对性补强，需要高质量的面经和实战题 |
| 行为特征 | 时间紧迫，目标明确，偏好"检测→补漏→再检测"的循环模式 |
| 时间投入 | 每天2-3小时，持续2-4周 |

#### 用户画像四：跨专业自学者

| 维度 | 描述 |
|------|------|
| 典型情况 | 非CS专业，出于兴趣或就业考虑自学编程 |
| 知识基础 | 无计算机基础，对算法、数据结构等计算机思维无概念 |
| 学习痛点 | 缺少计算机专业素养；不知道学编程的正确方法论；碰到问题求助无门 |
| 核心需求 | 需要完整的、系统的、带答疑的自学体系 |
| 行为特征 | 学习时间不稳定，动机强烈但容易受挫放弃 |

### 2.2 核心痛点提炼

| 编号 | 痛点 | 严重程度 | 当前状况 | 期望状态 |
|------|------|----------|----------|----------|
| P1 | 资源筛选困难 | 高 | 主动搜索B站/慕课/GitHub，靠运气碰到适合自己的内容 | 系统根据画像自动匹配和生成，一打开就是合适的内容 |
| P2 | 学习路径不清晰 | 高 | 跟着课本目录走，不知道哪些重要哪些可跳过 | 基于知识图谱和画像的个性化路径，动态调整 |
| P3 | 缺乏即时反馈 | 高 | 做完题看答案，对错知道了，但不知道为什么 | 每道题有解析，系统分析错误模式，推送针对性练习 |
| P4 | 难以评估自身水平 | 中 | 只有考试分数，不知道自己哪个模块弱 | 6维度画像实时更新，知识覆盖热力图一目了然 |
| P5 | 求助渠道单一 | 中 | 百度/StackOverflow/问同学，答案质量不可控 | 实时对话答疑，基于知识库的准确回答 |
| P6 | 资料形式单一 | 中 | 大部分是视频或文字，缺乏交互性 | 文档+练习题+可运行代码+思维导图+动画多形态 |

### 2.3 典型使用场景

**场景一：新生入门自学**

> 大一新生小明选修了JavaScript课程，在系统注册后进入对话。
>
> 系统（ProfileInterviewer）："你之前接触过编程吗？" 
> 小明："没有。"
> 系统："你学JavaScript是想拿学分，还是想以后做网页开发？"
> 小明："想做网页，最好能做出一个个人主页。"
>
> 经过8-10轮对话，系统构建了小明初始画像：零基础、实用型目标、偏好动手实践、每周可投入6小时。系统规划路径：HTML基础 → CSS基础 → JS变量/类型 → 函数 → DOM操作 → 事件 → 综合小项目。
>
> 小明开始学习。每学完一个节点，系统推送2-3道自适应练习题。小明的错误集中在"变量提升"概念上，系统自动追加一个变量提升的动画演示和3道对比练习。两周后小明完成"综合小项目"节点，做出了自己的个人主页。

**场景二：跨语言学生定点突破**

> 大二学生小红学过C语言，JavaScript总是搞不清this和闭包。她进入对话说明情况。
>
> 系统通过知识图谱分析发现：小红不理解JavaScript的作用域链和执行上下文，这是理解this和闭包的前置知识。系统在图谱上标注出缺失节点，先给她补了"作用域链"内容，然后生成了一份"this绑定规则"的图文文档（从C的指针类比切入），配5道this专项练习。
>
> 小红做完练习正确率60%，系统识别她容易在"隐式绑定"和"new绑定"上混淆，追加了对比练习。第二轮正确率提升到90%。系统更新画像，标注"this"已掌握，解锁下一个节点。

**场景三：面试冲刺复习**

> 大三学生小刚即将参加校招，需要在两周内快速复习JavaScript核心知识。他设定了每天2小时的时间投入。
>
> 系统根据画像（基础扎实，ES6+和异步需要加强），生成了复习路径：先做一套诊断题 → 跳过已掌握节点 → 聚焦薄弱项。每天推送当日学习清单：快速回顾文档（5分钟）+ 高频面试题（3道）+ 手写代码练习（1道）。系统追踪完成情况，每天调整次日计划。

## 3. 功能性需求

### 3.1 对话式学习画像构建（核心模块，优先级P0）

**需求描述**：系统通过自然语言对话收集学生信息，自动提取特征，构建并维护一个包含不少于6个维度的动态学生画像。

**6个画像维度定义**：

| 维度 | 含义 | 取值示例 | 数据来源 |
|------|------|----------|----------|
| 知识基础 | 对各知识点的掌握程度 | `{"变量": "熟练", "闭包": "未掌握", "异步": "了解"}` | 对话判断 + 做题正确率 |
| 认知风格 | 学生学习方式的偏好 | `视觉型 / 听觉型 / 动手型 / 理论型 / 混合型` | 对话中询问 + 资源使用行为 |
| 常见错误模式 | 学生反复犯错的知识点类型 | `[{"pattern": "混淆let/const作用域", "topic": "变量", "频率": 5}]` | 答题记录分析 |
| 学习偏好 | 喜欢的资源类型和学习节奏 | `{"资源类型": ["代码案例", "视频"], "节奏": "慢→快"}` | 对话 + 资源点击/完成率 |
| 学习目标 | 短期和长期目标 | `[{"目标": "做出Todo List", "优先级": "高", "截止": "2026-06-15"}]` | 对话收集 |
| 时间投入 | 可投入的学习时间和频率 | `"每周5小时, 集中在晚上"` | 对话收集 |

**功能子项**：

| 编号 | 功能点 | 说明 |
|------|--------|------|
| F1.1 | 对话式信息采集 | 系统发起引导式对话，每次聚焦1-2个维度，渐进式收集信息。避免一次性"审问式"提问 |
| F1.2 | 画像自动提取与更新 | 对话内容经LLM分析后自动提取关键信息，更新画像各维度。画像随学习行为持续更新 |
| F1.3 | 画像置信度计算 | 每个维度标注置信度（0-1）。对话直接确认的信息置信度高（>0.8），推断出的信息置信度低（<0.5） |
| F1.4 | 画像版本管理 | 画像每次更新生成新版本，保留历史。前端展示画像变化时间线 |
| F1.5 | 画像可视化展示 | 前端以6张卡片展示画像各维度：知识热力图、认知风格雷达图、错误标签云、偏好饼图、目标进度条、时间日历热力图 |

### 3.2 多智能体协同资源生成（核心模块，优先级P0）

**需求描述**：由不同角色的智能体组成资源生成流水线，协同完成至少5种类型个性化资源的生成、审核和输出。必须体现"多智能体"的协作关系。

**智能体角色与职责**：

| 角色 | 对应Agent | 职责 | 协作关系 |
|------|-----------|------|----------|
| 资源规划师 | ResourcePlanner | 分析画像和需求，决策生成哪些资源、什么难度、多少量 | 管理者，向其他Agent派发任务 |
| 文档撰写师 | DocGenerator | 生成适配学生水平的Markdown讲解文档 | 接收规划师的任务，交付给审核员 |
| PPT制作师 | PPTGenerator | 生成知识点教学PPT讲义 | 同上 |
| 出题专家 | ExerciseGenerator | 生成选择题/填空题/编程题，难度匹配学生水平 | 同上 |
| 案例设计师 | CodeCaseGenerator | 设计真实场景的代码实战案例，确保代码可运行 | 同上 |
| 项目设计师 | ProjectDesigner | 设计实践项目学习材料（需求说明+启动代码+测试用例+分步指引） | 同上 |
| 知识结构化师 | MindMapGenerator | 生成知识结构化的思维导图数据（JSON格式） | 同上 |
| 拓展阅读师 | ReadingGenerator | 生成与知识点相关的深入拓展阅读材料 | 同上 |
| 动画脚本师 | VideoAnimator | 生成教学动画的场景脚本（场景描述+旁白+动画指令） | 同上 |
| 质量审核员 | ResourceReviewer | 审核所有生成内容的准确性、安全性、适配度，不通过则打回重做 | 审核者 |

**8种资源类型**（赛题列举的资源类型全覆盖）：

| 编号 | 资源类型 | 赛题原文对应 | 输出格式 | 是否多模态 |
|------|----------|-------------|----------|------------|
| R0 | 资源设计方案 | **资源设计方案** | JSON（资源规划清单：类型决策、难度策略、知识点覆盖范围） | 否 |
| R1 | 课程讲解文档 | 专业课程讲解文档 | Markdown（含代码块） | 否 |
| R2 | 知识点思维导图 | 知识点思维导图 | JSON（前端渲染为交互式导图） | 是 |
| R3 | 练习题集 | 不同类型练习题目 | JSON（前端渲染为Web答题组件） | 是 |
| R4 | 代码实战案例 | 实操案例 | Markdown + 可执行代码 | 是 |
| R5 | PPT讲义 | **PPT** | PPTX文件 / Markdown转PPT | 是 |
| R6 | 实践项目材料 | **实践项目学习材料** | JSON（需求说明+启动代码+测试用例+分步指引） | 是 |
| R7 | 拓展阅读材料 | 拓展阅读材料 | Markdown（含外部参考链接） | 否 |
| R8 | 教学动画脚本 | 多模态教学视频/动画 | JSON（含场景/旁白/动画指令） | 是 |

> 注：R0资源设计方案是ResourcePlanner在每次资源生成前的规划输出，作为元资源记录学生每次生成请求的决策依据，便于追溯和优化。

**资源生成协作流程**：

```
1. 学生触发资源生成请求（指定知识点 + 附加要求）
2. ResourcePlanner 分析：画像 + 知识点图谱 → 生成资源设计方案(R0)，确定资源组合、难度、范围
3. ResourcePlanner 向各专业Agent派发任务（可并行）
      ├── DocGenerator → 生成讲解文档(R1)
      ├── PPTGenerator → 生成PPT讲义(R5)
      ├── ExerciseGenerator → 生成练习题(R3)
      ├── CodeCaseGenerator → 生成代码案例(R4)
      ├── ProjectDesigner → 生成项目材料(R6)
      ├── MindMapGenerator → 生成思维导图(R2)
      ├── ReadingGenerator → 生成拓展阅读(R7)
      └── VideoAnimator → 生成动画脚本(R8)
4. ResourceReviewer 逐一审核：
      ├── 通过 → 标记已通过，入库
      └── 不通过 → 带修改意见返回生成Agent，最多重试2次
5. 审核完成的资源推送到前端
```

**功能子项**：

| 编号 | 功能点 | 说明 |
|------|--------|------|
| F2.1 | 资源类型智能决策 | 根据学生画像和请求内容，自动判断应生成哪些类型的资源（8种中选取适合的组合，不需要每次都生成全部） |
| F2.2 | 难度自适应 | 生成的资源难度匹配学生当前水平。新生偏基础，进阶生偏深入 |
| F2.3 | 知识库约束生成 | 所有生成内容必须基于RAG检索的知识库片段，不允许编造 |
| F2.4 | 生成进度实时展示 | 前端流式展示：当前哪个Agent在工作、生成了什么内容、审核状态 |
| F2.5 | 代码执行验证 | 代码案例生成后，自动在Deno沙箱中执行验证，确保可运行 |

### 3.3 知识图谱驱动的学习路径规划（核心模块，优先级P0）

**需求描述**：基于知识图谱的图算法，结合学生画像，为每位学生生成动态、最优的个性化学习路径。

**知识图谱结构**：
- 节点：85个JavaScript知识点，每个节点包含：标题、难度等级（初/中/高）、详细内容Markdown、向量嵌入
- 边：知识点间的依赖关系（prerequisite前置/related相关/extends延伸/contrasts对比）

**图算法应用**：

| 算法 | 用途 | 场景 |
|------|------|------|
| 拓扑排序 | 确定合法学习顺序 | 所有学生都需要按依赖关系排序 |
| Dijkstra最短路径 | 计算从当前水平到目标的最优路径 | 学生设定了明确的学习目标 |
| PageRank | 识别枢纽知识点（影响多个下游节点的关键知识） | 优先推荐重要性高的节点 |
| 断点检测 | 识别知识链中的断裂点 | 学生某个概念理解不了，回溯找前置缺失 |

**功能子项**：

| 编号 | 功能点 | 说明 |
|------|--------|------|
| F3.1 | 个性化路径生成 | 输入学生画像 + 学习目标，输出排序后的学习节点列表，每节点标注：序号、状态（锁定/就绪/学习中/已完成）、推荐资源、预估时长 |
| F3.2 | 路径可视化 | 前端以时间线+3D知识图谱两种视图展示学习路径。3D图中节点大小表示PageRank值，颜色表示掌握程度（红=未学→黄=学习中→绿=已掌握），路径以发光连线高亮 |
| F3.3 | 路径动态调整 | 学生学习过程中，系统根据新的答题数据、资源使用行为重新评估画像，当画像变化超过阈值时触发路径重新规划 |
| F3.4 | 资源精准推送 | 每个路径节点自动关联匹配难度的推荐资源。学生进入节点后优先展示已生成资源+可触发生成新资源 |
| F3.5 | 断点诊断 | 当学生在某个知识点反复出错时，系统在图谱上回溯前置节点，标注可能的断裂点并推送补充学习内容 |

### 3.4 智能辅导（可选加分项，优先级P2）

**需求描述**：学生在学习过程中遇到问题时，系统提供即时答疑。解答形式不限于文字，鼓励多模态（图解、步骤动画等）。

| 编号 | 功能点 | 说明 |
|------|--------|------|
| F4.1 | 实时对话答疑 | 学生可以在对话中随时提问，系统基于知识库检索+LLM推理给出准确回答 |
| F4.2 | 多模态解释 | 对复杂概念（如事件循环、闭包），系统生成分步图解描述或动画脚本 |
| F4.3 | 答疑关联画像更新 | 学生提问的内容自动关联画像更新（如反复问相似概念，标记为该维度薄弱） |

### 3.5 学习效果评估（可选加分项，优先级P2）

**需求描述**：追踪学生行为数据，多维度评估学习效果，驱动推荐策略优化。

| 编号 | 功能点 | 说明 |
|------|--------|------|
| F5.1 | 学习行为追踪 | 记录：页面停留时长、资源打开/完成、答题正确率/耗时、主动提问内容 |
| F5.2 | 多维度评估 | 评估指标：知识掌握度、学习投入度、进步速度、薄弱点收敛趋势 |
| F5.3 | 自适应调整 | 根据评估结果自动调整：路径节点顺序、资源推荐策略、推送频率 |

## 5. 前沿AI技术与需求的结合点

这是赛题明确要求分析的部分：**AI技术如何解决具体的用户需求问题**。

| 技术 | 解决什么问题 | 结合方式 | 对应功能 |
|------|-------------|----------|----------|
| **大语言模型(LLM)** | 学生需要个性化问答，传统系统只能返回预设答案 | 讯飞星火4.0 Ultra作为Agent的推理引擎，理解学生意图、生成个性化内容 | 对话画像、资源生成、答疑 |
| **多智能体框架(CrewAI)** | 单一AI无法胜任多重角色（既要当老师又要当出题人），需要专业化分工 | 定义不同角色的Agent，各司其职，通过Hierarchical Process协作 | 资源生成流水线 |
| **RAG检索增强生成** | LLM有"幻觉"问题，编造不存在的知识点 | 所有Agent生成前强制检索知识库，基于检索内容生成，禁止自由发挥 | 所有资源生成 |
| **知识图谱+图算法(NetworkX)** | 线性课程无法适配不同学生的知识起点和盲区 | 构建JS知识有向图，用拓扑排序/最短路径/PageRank算法计算最优路径 | 学习路径规划 |
| **多模态大模型(SeeDance/讯飞PPT)** | 纯文字教程无法演示代码执行过程等动态概念；赛题明确要求生成PPT和教学动画 | 调用SeeDance生成教学动画，调用讯飞星火PPT API生成教学讲义，前端Canvas渲染代码流程动画 | 教学动画(R8)、PPT讲义(R5) |
| **语音合成(TTS)** | 阅读疲劳，听觉通道未被利用 | 将讲解文档转语音，支持"听课"模式；实践项目材料可与语音讲解同步 | 多模态资源 |
| **代码沙箱(Deno)** | 生成的代码示例是否正确，学生照着写可能报错 | 生成后自动在沙箱执行验证，不通过的代码打回重生成。实践项目材料的测试用例也在沙箱中验证 | 代码案例(R4)、项目材料(R6) |
| **内容审核API** | 防止生成违规内容 | 讯飞内容审核API对最终输出做最后一道把关 | 安全过滤 |
| **SSE流式输出** | 学生等待时长焦虑，长文本一次性加载体验差 | Agent输出逐Token推送到前端，打字机效果实时渲染 | 全流程交互 |

### 创新价值总结

| 创新点 | 传统方案 | 本系统方案 | 用户价值 |
|--------|----------|------------|----------|
| 学习路径 | 固定课表/目录 | **图算法驱动的动态路径** | 学得科学、不走弯路 |
| 学习内容 | 统一教材/视频 | **基于画像的生成式自适应内容** | 每个人看到的都是适合自己的版本 |
| 内容形式 | 单一视频/文字 | **8种类型多模态输出**（文档+PPT+导图+练习+代码+项目+阅读+动画） | 视觉型/动手型/理论型学生各取所需 |
| 智能分工 | 一个AI干所有事 | **10个专业Agent协同**（规划/文档/PPT/练习/代码/项目/导图/阅读/动画/审核） | 专业的事交给专业的Agent做，质量更高 |
| 质量保障 | 人工审核或干脆不审核 | **多层自动审核+代码沙箱验证** | 更可靠、更安全 |

## 7. 约束与假设

### 约束

1. 讯飞星火API需可访问，且稳定性满足开发测试要求
2. 开发周期12周，团队人手和技能需覆盖前后端+AI集成

2. 假设初始知识库质量足够支撑RAG生成准确内容
3. 假设Demo阶段不与真实教学系统集成，使用构造的测试学生数据
4. 多模态生成API可用性待验证，若不可用则降级为前端交互渲染方案
