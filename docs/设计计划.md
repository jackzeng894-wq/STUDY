# 基于大模型的个性化资源生成与学习多智能体系统 - 设计计划

## Context

参加第十五届中国软件杯大赛A组赛题，需从零构建一个多智能体学习系统。通过CrewAI多智能体协作，为JavaScript课程生成个性化学习资源，实现"因材施教"的数字化落地。

评分权重：创新实用性35% + 功能实现45% + 文档10% + 演示10%

---

## 技术选型

| 层 | 技术 | 理由 |
|---|---|---|
| 后端 | Python + FastAPI | 异步支持好，AI生态完善 |
| 前端 | Vue 3 + TypeScript + Vite + Naive UI | 国内生态好，中文组件开箱即用 |
| 多智能体 | CrewAI | 角色驱动，天然适配不同Agent分工 |
| LLM | 讯飞星火4.0 Ultra (OpenAI兼容接口) | 赛题要求使用讯飞工具 |
| 多模态生成 | SeeDance / 讯飞星火PPT / 前端动画渲染 | 赛题明确要求多模态视频/动画生成 |
| 知识图谱 | NetworkX + Three.js 3D力导向图 | 图算法驱动路径规划，创新亮点 |
| 数据库 | PostgreSQL + pgvector | 关系数据+向量检索一体化 |
| 流式输出 | SSE (Server-Sent Events) | FastAPI原生支持，比WebSocket更简单 |

## 软件工程设计理念

### 1. 架构决策：模块化单体

约束分析：
- 开发周期最多12周，微服务的服务发现、容错、分布式追踪成本占据开发时间
- 团队规模大概率是1-3人，微服务运维负担不成比例
- 比赛演示环境只需撑住评委操作，不需要弹性伸缩

**决策：模块化单体（Modular Monolith）**

```
┌─────────────────────────────────────────────────┐
│                  FastAPI 应用                      │
│                                                   │
│  api/          ← 路由层，薄层，只做参数校验和调用 │
│  services/     ← 业务逻辑，编排各模块            │
│  agents/       ← CrewAI智能体定义                │
│  rag/          ← 检索增强生成管线                │
│  iflytek/      ← 外部API适配                     │
│  streaming/    ← SSE事件总线                     │
│  models/       ← ORM，数据访问                   │
│                                                   │
│  每个目录有清晰的接口边界，但不跨进程通信          │
└─────────────────────────────────────────────────┘
```

核心约束：**模块间通过Service层调用，不直接跨模块引用**。将来如果某个模块成为瓶颈，只需拆出去加一层HTTP适配，内部逻辑不用改。

### 2. 后端四层分层设计

```
┌──────────────────────────────────────┐
│  API 层 (api/)                       │  ← 参数校验、认证、响应格式化
│  只做：接收请求 → 校验 → 调用Service │     不做业务逻辑
├──────────────────────────────────────┤
│  Service 层 (services/)              │  ← 业务编排
│  做：编排多个模块完成一个业务场景     │     例如 resource_service.py 负责
│      触发Agent → 监听事件 → 存储资源 │     "生成资源"的完整流程
├──────────────────────────────────────┤
│  Domain 层 (agents/ + rag/)          │  ← 核心领域逻辑
│  做：Agent定义、工具定义、RAG检索    │     不关心HTTP、不关心数据库
├──────────────────────────────────────┤
│  Infrastructure 层 (models/ +        │  ← 基础设施
│      iflytek/ + streaming/)          │     数据库、外部API、消息推送
└──────────────────────────────────────┘
```

**依赖规则**：上层可以调用下层，下层绝不引用上层。Domain层不依赖Infrastructure的具体实现。

| Service | 职责 | 核心方法 |
|---------|------|----------|
| `profile_service` | 画像对话编排 | `start_profile_conversation()` → 触发ProfileCrew → 解析Agent输出 → 更新画像 |
| `resource_service` | 资源生成编排 | `generate_resources()` → 触发ResourceCrew → 监听SSE事件 → 存储通过审核的资源 |
| `path_service` | 路径规划编排 | `generate_path()` → 调用graph_service算图 → 触发PathCrew → 组装最终路径 |
| `safety_service` | 内容安全 | `verify_resource()` → RAG一致性检查 → 代码沙箱 → 讯飞审核 |
| `graph_service` | 图算法 | `topological_sort()`, `shortest_path()`, `pagerank()`, `detect_gaps()` |

### 3. 多智能体编排的三个核心设计

#### 3.1 Agent间数据结构化

不用字符串拼接传递上下文，用结构化对象：

```python
@dataclass
class ResourceTask:
    task_id: str
    resource_type: str           # "course_doc" | "exercise" | "code_case" | ...
    knowledge_node_id: str
    difficulty: str              # "beginner" | "intermediate" | "advanced"
    student_profile_snapshot: dict  # 生成时的画像快照（用于追溯）
    rag_context: str             # RAG检索到的知识库片段（注入Agent防幻觉）
    constraints: list[str]       # ["代码必须可运行", "难度不超intermediate"]
```

好处：
- 可追溯：每次生成的输入参数完整记录，方便排查"为什么生成得不好"
- 可测试：可以单独构造一个ResourceTask来测试某个Agent
- 类型安全：不会把错误的参数传给Agent

#### 3.2 并行生成 + 顺序审核

```
ResourcePlanner
       │
       ├──→ DocGenerator ──────────┐
       ├──→ PPTGenerator ──────────┤
       ├──→ ExerciseGenerator ─────┤
       ├──→ CodeCaseGenerator ─────┼──→ ResourceReviewer → 推送给用户
       ├──→ ProjectDesigner ───────┤         │
       ├──→ MindMapGenerator ──────┤    不通过 → 打回重生成（最多2次）
       ├──→ ReadingGenerator ──────┤
       └──→ VideoAnimator ─────────┘
```

CrewAI的Hierarchical Process天然支持这个模式。关键处理：**如果某个Agent耗时过长（如VideoAnimator调用外部API），已完成的资源先审核先推送，不必等全部完成**。

#### 3.3 分级降级策略

| 失败场景 | 策略 | 用户体验 |
|----------|------|----------|
| 单个Agent超时 | 重试1次 → 仍失败则跳过该资源类型 | 提示"XX资源暂时跳过，其他资源已生成" |
| Reviewer打回 | 带修改意见重试，最多2次 → 仍不通过则标记"人工审核" | 资源标记黄色叹号"内容待确认" |
| 讯飞API不可用 | 降级为本地规则过滤（敏感词库） | 透明降级，用户无感知 |
| 代码沙箱执行失败 | 标记代码块"未验证"，但仍展示 | 代码块上灰色提示"此代码未经沙箱验证" |

### 4. 数据架构设计

#### 4.1 pgvector选择理由

| 因素 | pgvector | 独立向量库(Milvus/Qdrant) |
|------|----------|---------------------------|
| 运维复杂度 | 0（PostgreSQL扩展，无额外进程） | 需要单独部署维护 |
| 85个知识节点规模 | 完全足够 | 杀鸡用牛刀 |
| 事务一致性 | 向量和元数据在同一事务中 | 跨系统需自己保证一致性 |
| 部署 | 一个Docker容器 | 多个Docker容器 |

在代码中抽象一层VectorStore接口，将来如果需要迁移，只改实现层。

#### 4.2 JSONB设计原则

8种资源类型结构不同，用JSONB存储内容，但遵循原则：**查询用的字段提到表层，展示用的内容放在JSONB**。

```sql
-- 表层：用于筛选、排序、关联
resource_type, title, knowledge_node_ids, difficulty, target_student_id, review_status
-- JSONB：具体内容，不同类型不同结构
content, generation_context
```

### 5. 前端事件驱动状态管理

对话页同时存在4个状态流：消息流（SSE逐token推送）、Agent状态、资源流、画像更新。如果放在组件local state里会失控。

**决策：事件驱动的Pinia Store**

```
SSE事件 → EventBus → Pinia Store (单一状态树) → 组件响应式渲染
```

```typescript
// 核心状态设计
interface ConversationState {
  messages: Message[]              // 所有消息（含流式追加中的消息）
  activeAgents: Map<string, AgentStatus>  // 当前活跃的Agent及其状态
  pendingResources: Resource[]     // 生成中/待审核的资源
  approvedResources: Resource[]    // 已通过审核的资源
  profileUpdates: ProfileDiff[]    // 本次对话中画像的变更
}
```

关键原则：**SSE事件只做一件事——dispatch到Store**。组件不直接监听SSE，这样测试时可mock Store，未来换WebSocket组件代码不用改。

### 6. 错误处理分级

10个Agent的系统，出错是常态，需分清楚什么错误给谁看：

| 错误层级 | 示例 | 用户看到什么 | 开发者看到什么 |
|----------|------|-------------|---------------|
| **L1 用户可恢复** | 生成参数不明确 | "请告诉我你想学哪个具体知识点？" | info日志 |
| **L2 系统可恢复** | Agent超时 | "XX资源暂时跳过，你稍后可以重新生成" | warning日志 + 指标 |
| **L3 需人工介入** | 审核多次不通过 | 资源显示黄色叹号 | error日志 + 数据库标记 |
| **L4 致命** | 数据库挂了 | "系统暂时不可用，请稍后重试" | critical告警 |

每一层错误有独立的处理路径，不混在一起。

### 7. 测试分层策略

12周项目不可能全覆盖测试，按投入产出比分层：

| 测试层级 | 投入 | 测试什么 | 不测试什么 |
|----------|------|----------|------------|
| **单元测试（重点投入）** | 必须有 | graph_service图算法（确定性逻辑，最适合单测）、JSONB序列化/反序列化、画像维度解析 | - |
| **Agent工具测试** | 应该有 | knowledge_tool检索返回格式、code_exec_tool沙箱执行 | Agent的LLM输出内容（不可控） |
| **集成测试** | 关键路径 | ProfileCrew完整流程、ResourceCrew单资源生成 | 8个Agent全部并行场景 |
| **E2E测试** | 不投入 | - | 自动化E2E成本太高，人工验收代替 |

核心测试原则：**把确定性逻辑（图算法、数据变换、API格式校验）和不确定性逻辑（LLM输出）严格分开测试**。不断言"LLM应该输出什么内容"，只断言"LLM的输出格式符合预期结构"。

### 8. 三个"不要"

1. **不要过度抽象**：10个Agent不需要10个Factory、10个Strategy。CrewAI本身的Agent和Task就是抽象，在其上再加层只会让调试变困难。
2. **不要提前优化**：85个节点的知识图谱不需要缓存预热、不需要分布式图计算。NetworkX在内存里跑完全够。
3. **不要假设API可靠**：讯飞星火、SeeDance、TTS——每个外部API都可能超时、可能返回格式变了、可能突然需要重新认证。每个外部调用都要有超时+重试+降级三层保护。

## CrewAI多智能体设计

### Crew 1: ProfileCrew (画像构建) - Sequential Process

| Agent | 角色 | 职责 | 工具 |
|-------|------|------|------|
| ProfileInterviewer | 学生学情访谈专家 | 自然语言对话收集学生信息 | profile_tool |
| ProfileAnalyzer | 学习者画像分析专家 | 结构化输出6维度画像 | profile_tool, knowledge_tool |

6维度：知识基础、认知风格、常见错误模式、学习偏好、学习目标、时间投入

### Crew 2: ResourceCrew (资源生成，核心) - Hierarchical Process

| Agent | 角色 | 职责 | 工具 |
|-------|------|------|------|
| ResourcePlanner (Manager) | 教学资源规划师 | 规划资源类型/难度，分配任务 | profile_tool, knowledge_tool |
| DocGenerator | 课程文档撰写专家 | 生成Markdown讲解文档(R1) | knowledge_tool, safety_tool |
| PPTGenerator | PPT制作师 | 生成知识点教学PPT讲义(R5) | knowledge_tool, safety_tool |
| ExerciseGenerator | 练习题设计专家 | 生成选择题/填空题/编程题(R3) | knowledge_tool, code_exec_tool |
| CodeCaseGenerator | 实战案例设计专家 | 生成可运行代码案例(R4) | knowledge_tool, code_exec_tool |
| ProjectDesigner | 项目设计师 | 设计实践项目学习材料(R6) | knowledge_tool, code_exec_tool |
| MindMapGenerator | 知识可视化专家 | 生成思维导图JSON数据(R2) | knowledge_tool |

## 知识图谱驱动的路径规划 (创新亮点)

### 架构流程

```
knowledge_nodes + knowledge_relations (PostgreSQL)
         ↓
   NetworkX 知识图谱构建
         ↓
   图算法分析 (拓扑排序/最短路径/PageRank/断点检测)
         ↓
   PathDesigner Agent + 图算法结果 → 个性化路径
         ↓
   Three.js 3D力导向图可视化
```

### 图算法应用

| 算法 | 用途 | 场景 |
|------|------|------|
| 拓扑排序 | 确定合法学习顺序 | 所有学生都需要按依赖关系排序 |
| Dijkstra最短路径 | 从当前水平到目标的最优路径 | 学生设定了明确的学习目标 |
| PageRank | 识别枢纽知识点 | 优先推荐重要性高的核心节点 |
| 断点检测 | 识别知识链中的断裂点 | 学生某个概念理解不了，回溯找前置缺失 |

### 前端3D知识图谱可视化

- 使用 Three.js + 3D力导向布局渲染知识图谱
- 节点大小 = PageRank值，颜色 = 掌握程度(红→未学，黄→学习中，绿→已掌握)
- 边 = 知识点依赖关系，高亮当前学习路径
- 支持拖拽旋转、点击节点查看详情
- 学习路径在图谱上以发光路径高亮显示

## 反幻觉与内容安全 (5层防护)

1. **RAG检索增强** - Agent生成前必须检索知识库，prompt强制引用检索结果，禁止自由发挥
2. **代码执行验证** - 生成的JS代码在Deno沙箱中执行验证，不通过打回重生成
3. **ResourceReviewer审核** - 专门Agent交叉验证内容准确性、代码可运行性、难度适配度
4. **讯飞内容审核API** - 对最终输出做最后一道安全把关
5. **引用溯源** - 所有资源标注知识来源，前端显示"信息来源"标签

## 流式输出实现

```
Agent Token → StreamingCallbackHandler → EventBus → SSE Manager → FastAPI SSE → Vue EventSource
```

- EventBus分发到各SSE订阅者
